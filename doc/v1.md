# v1.0 — Folding, Clipboard, DIGRAPH Links

Post-MVP milestone. Branch: `v1.0-digraph`.

## Goal

Extend the outliner from a pure tree to a DIGRAPH (directed acyclic graph)
where a single node can appear under multiple parents via link-paste.
This requires three prerequisite capabilities delivered in sequence.

## Data Model Changes

### SiteNode (render tree)

The MVP renders directly from `Graph` — a full recursive walk on every
render. v1.0 introduces `SiteNode`, a persistent render tree that:

- Assigns each **appearance** of a node a stable `instanceId: int`
- Tracks per-appearance `expanded: bool` (fold state)
- Allows the same `NodeId` to appear in multiple locations with
  independent fold states
- Is reconciled (not rebuilt) after graph mutations, preserving
  existing instance IDs

```
type SiteNode =
    { instanceId: int
      nodeId: NodeId
      expanded: bool
      children: SiteNode list }
```

`Model` gains `siteRoot: SiteNode` and `nextInstanceId: int`.

### ClipboardContent

Internal clipboard holds a self-contained snapshot of copied/cut nodes,
independent of `graph.nodes`:

```
type ClipboardContent =
    { topLevelIds: NodeId list
      nodes: Map<NodeId, Node> }
```

`Model` gains `clipboard: ClipboardContent option`.

#### 3c-i. Flat SiteMap — New Types and Functions (additive)

The `siteRoot: SiteNode` nested tree is replaced with a flat `SiteMap` that mirrors `Graph`
in the same way `Graph` mirrors the data model. This enables O(log S) per-entry access and
is the prerequisite for the incremental DOM renderer (3d) and link-paste (3e).

```
type SiteEntry =
    { instanceId: int
      nodeId: NodeId
      parentInstanceId: int option   // None = root
      expanded: bool
      children: int list }           // instanceId list, ordered to match graph.children

type SiteMap =
    { rootId: int
      entries: Map<int, SiteEntry> }
```

**Complexity (S = site entries, V = visible, N = graph nodes, C = child count):**

| Operation | Before | After |
|---|---|---|
| Occurrence lookup | O(S) scan | O(log S) |
| `toggleFold` collapse | O(S) scan | O(log S) |
| `toggleFold` expand | O(S) + O(N) rebuild | O(log S) + O(C log S) |
| Structural op (indent/cut/paste) | O(N) full rebuild | O(N log N) reconcile |
| Load / `StateLoaded` | O(N) | O(N log N) — one-time |

New types and functions added alongside the existing `SiteNode` infrastructure; app
continues to run on the old path. No callsite changes.

#### 3d. Incremental DOM Renderer

`render` in `View.fs` currently calls `app.innerHTML <- ""` on every dispatch, destroying and
recreating every visible DOM node. At 100K nodes with 5K visible that is ~5K `createElement`
calls per keypress while editing — the dominant cost, independent of the SiteMap design.

The flat SiteMap makes incremental patching practical: a parallel `Map<int, HTMLElement>` element
cache uses the same `instanceId` key, so any changed `SiteEntry` maps directly to its DOM row.

- [x] Add `elementCache: Map<int, HTMLElement>` as a mutable alongside `currentModel` in `Program.fs`
- [x] Replace the full-rebuild `render` with `patchDOM : Model -> Model -> Map<int, HTMLElement> -> Map<int, HTMLElement>`
      that diffs old and new `SiteMap` and issues only necessary DOM mutations:
  - Entry removed → `element.remove()`, remove from cache
  - Entry added → `createElement`, insert at correct sibling position, add to cache
  - `expanded` changed to false → remove child DOM rows from under this entry; update cache
  - `expanded` changed to true → insert new child DOM rows; update cache
  - Graph text changed for a `nodeId` → `textDiv.textContent <- newText` on each cached element
    whose `nodeId` matches (O(K) where K = visible occurrences); no node creation
  - Selection or focus class changed → update `className` on affected rows only
- [x] Full rebuild retained for `StateLoaded` only (load-time, acceptable O(V))
- [x] End-to-end per-operation DOM cost:
  - `SetText`: O(K) `.textContent` assignments — no node creation
  - `ToggleFold` expand: O(C) `insertBefore` calls
  - `ToggleFold` collapse: O(C) `remove` calls
  - Structural op: O(diff) insertions/removals proportional to what changed
- [x] Unit tests: `planPatchDOM` text change produces `SetText` patch and no `CreateRow`;
      expand produces correct `CreateRow` count; collapse produces correct `RemoveRow` count

## Key Design Decisions

| Decision | Rationale |
|---|---|
| Flat `SiteMap` keyed by `instanceId` | O(log S) per-entry access; enables incremental DOM patching; same design as `Graph`; each occurrence is an independent entry even if two share a `nodeId` |
| `ClipboardContent` is a standalone snapshot | Independent of `graph.nodes`; survives graph mutations and snapshot reload |
| `copy`/`cut` event handlers instead of key table | `e.clipboardData.setData` is synchronous and permission-free; works on non-HTTPS and Firefox; no key table entries needed for `Ctrl+C`/`Ctrl+X` |
| `serializeSubtree` called in View handler, not `update` | `e.clipboardData` is only valid synchronously during the event; must write before handler returns |
| Single `Ctrl+V` for paste; settings option controls link vs deep-copy | Avoids `navigator.clipboard.readText()` (async, HTTPS-only, permission-gated); no custom keybinding needed |
| Deep-copy = fresh NodeIds, link-paste = reuse NodeIds (Phase 3) | Link-paste is the more common intent in a DIGRAPH outliner; deep-copy is the default |
| Snapshot format extended with `#id` / `-> #id` | Human-readable, backward compatible with plain tab-indented files |
| Cycle guard in `reconcileSiteMap`, per-path ancestor set | Graph can be cyclic; guard is mandatory, not defense-in-depth; per-path (not global visited) so a node in two branches is permitted in both |
